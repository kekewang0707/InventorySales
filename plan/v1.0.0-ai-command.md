# v1.0.0 — AI 快捷操作

## 目标

用户通过自然语言（文本输入或语音转文字）描述需求，AI 理解语义并自动执行操作或给出应答。

当前版本范围：查询类操作 + 少量受控写入（创建送货单需要二次确认）。后续版本再扩大到全量 CRUD，以及基于历史数据的数据分析问答。

---

## 1) 架构 — ReAct 模式（Reason + Act）

核心思路：LLM 可以自行决定"思考 → 调用工具 → 观察结果 → 再思考"的循环，直到给出最终回答。

```
[前端] AI 浮窗输入文本
  ── POST /api/ai/command { text: "..." } ──→ [后端]

  ReAct 循环（最多 5 轮）:
  ┌──────────────────────────────────────────────────────┐
  │  ① Reason: LLM 收到消息 → 选择工具或直接回复        │
  │        │                                            │
  │        ├── ② 查询类工具 → 执行 → 结果喂回 LLM ──→ ① │
  │        │              (Observe → Reason)             │
  │        ├── ② 写入类工具 → 暂存 → 返回 needs_confirm │
  │        │              ↓ 用户确认                     │
  │        │         POST /api/ai/confirm                │
  │        │              ↓                              │
  │        │         执行写入 → 结果喂回 LLM → ①         │
  │        │                                            │
  │        └── ③ LLM 返回文本 → 结束循环 → 返回 queried │
  └──────────────────────────────────────────────────────┘
```

**ReAct 模式的优点**:
- 多步推理：用户说"查一下客户 A 这个月的送货单金额最高的那笔"，LLM 先查客户 → 查送货单 → 筛选金额最高的 → 组织回答
- 上下文连贯：中间结果都留在对话中，LLM 可以引用之前的信息
- 降级自然：LLM/网络不可用时自动降级到离线规则引擎

**技术栈**:
- LLM: OpenAI Chat Completions API (gpt-4o-mini, 低成本/低延迟)，通过 Function Calling 返回结构化指令
- 后端: 新 `ai_service.py` 调用已有 service 函数；新 `routers/ai.py` 处理会话 state
- 前端: Element Plus Dialog 浮窗 + 确认框
- 配置: 通过 `backend/config.py` 新增 OpenAI API Key、Model Name、Base URL 等配置项，通过环境变量 `OPENAI_API_KEY` 等读取

## 2) LLM Function Calling 工具

LLM 注册以下 Function（OpenAI function calling 格式）。后端接收 LLM 返回的 `function_call` 后，映射到对应 service 函数执行。

### 查询类（直接执行）

| Function Name | 参数 | 后端映射 | 说明 |
|--------------|------|---------|------|
| `search_product` | `query: string`, `page?: int` | `product_service.list_products(search=query, page=page)` | 按名称/型号搜索产品 |
| `search_customer` | `query: string`, `page?: int` | `customer_service.list_customers(search=query, page=page)` | 按名称/电话搜索客户 |
| `get_customer` | `customer_id: int` | `customer_service.get_customer(customer_id)` | 查看客户详情 |
| `get_product` | `product_id: int` | `product_service.get_product(product_id)` | 查看产品详情 |
| `list_delivery_notes` | `customer_id?: int`, `start_date?: string`, `end_date?: string`, `page?: int` | `delivery_service.list_notes(...)` | 查询送货单列表，日期格式 YYYY-MM-DD |
| `get_delivery_note` | `note_id: int` | `delivery_service.get_note(note_id)` | 查看送货单详情（含明细行、产品信息） |
| `get_statement` | `customer_id?: int`, `start_date: string`, `end_date: string` | `routers/statements._query_notes()` + `_build_statements()` | 查询对账单，支持全部客户 |
| `get_recent_logs` | `entity_type?: string`, `action?: string`, `limit?: int` | `audit_service.list_logs(page_size=limit)` | 查看最近操作日志 |
| `get_statistics` | 无参数 | 聚合查询（产品总数、客户总数、送货单总数、本月总金额） | 查看系统统计概览 |

### 写入类（需二次确认）

| Function Name | 参数 | 后端映射 | 说明 |
|--------------|------|---------|------|
| `create_delivery_note` | `customer_id: int`, `delivery_date: string`, `items: [{product_id, quantity, unit_price}]`, `remark?: string` | `delivery_service.create_note(...)` | 创建送货单，暂存为 draft 状态 |
| `advance_note_status` | `note_id: int` | `delivery_service.advance_status(note_id)` | 推进送货单状态 draft→saved→reviewed |

### Function Call 参数约束

由于 OpenAI function calling 会把 LLM 认为的默认值自动填充，在 function definition 中应谨慎设置 `required` 字段。例如 `search_product` 的 `query` 和 `page` 均非必需（page 默认 1，query 为空时返回全部产品），但 LLM 通常需要 query 才能有效搜索。建议将所有参数标记为 非强制 required，在后端做默认值补偿。

## 3) API 端点

- `POST /api/ai/command`
  - Request: `{ "text": "..." }`
  - Response:
    - 查询成功: `{ "action": "queried", "reply": "今天有 3 张送货单...", "confirm_id": null, "tool_calls": null }`
    - 需确认: `{ "action": "needs_confirm", "reply": "即将创建送货单：客户XXX，金额XXX元", "confirm_id": "uuid-string", "tool_calls": [...] }`
    - 失败: `{ "action": "error", "reply": "抱歉，无法理解您的指令...", "confirm_id": null, "tool_calls": null }`
  - 流程:
    1. 后端收到 text，组装 system prompt（含今日日期） + function definitions 调 LLM
    2. LLM 返回 function_call（或纯文本回复/对话回复）
    3. 查询类 → 立即执行 service 函数，组合 reply 返回
    4. 写入类 → 暂存到内存 session（keyed by confirm_id），返回 `needs_confirm`
    5. API Key 未配置或 LLM 调用失败 → 降级到离线规则引擎
- `POST /api/ai/confirm`
  - Request: `{ "confirm_id": "uuid-string" }`
  - Response: `{ "action": "executed", "reply": "已成功创建送货单，编号 XXX" }`
  - 成功后清除 session 中的 pending action

### 会话状态

`ai_service.py` 维护一个内存 dict `pending_actions: dict[str, PendingAction]`，key 为 uuid 字符串，5 分钟过期 TTL。应用重启后清空（可接受）。

```python
@dataclass
class PendingAction:
    tool_name: str
    params: dict
    text_summary: str          # 用于前端确认框展示"即将执行：创建送货单..."
    db_session: Any            # FastAPI Depends 传入的 db session（用 weakref 或序列化规避）
    created_at: datetime
```

> 注意：`db_session` 跨请求传递需谨慎。方案是 `PendingAction` 只存 tool_name + params，确认时由 `confirm` 端点重新获取 db session。

## 4) System Prompt 设计

```txt
你是一个工厂销售出库管理系统的 AI 助手，系统名为 InventorySales。
用户可以查询产品、客户、送货单、对账单、操作日志等。
请根据用户的问题选择最合适的函数，或直接回答。

当前日期：{today}

核心约束：
- 查询类（search_*, get_*, list_*）→ 直接调用函数并返回结果给用户。
- 写入类（create_*, advance_*）→ 先返回操作摘要，引导用户确认。不要直接执行。
- 日期格式统一使用 YYYY-MM-DD。
- 货币单位是元（CNY），保留两位小数。
- 当用户问"今天"时，使用当前日期 {today}；"最近"默认为最近 7 天。
- 当搜索结果为空时明确告知用户。

回复时用中文，语言简洁自然，把关键信息（数字、名称、金额）放在前面。
对列表类结果做简要汇总，不要逐条朗读全部数据。
```

system prompt 中的 `{today}` 由后端每次请求时通过 `datetime.now().strftime("%Y-%m-%d")` 动态注入。

## 5) 安全设计

- **查询类操作**：直接执行，无额外确认
- **写入类操作**：LLM 返回 function_call → 暂存到内存 session → 前端弹确认框 → 用户点击确认 → 执行
- **所有 AI 操作**均记录 audit log，entity_type 为 `"ai_command"`，记录 `user_text`、`tool_call`（JSON）、`result`（摘要）
- **Confirmation session** 5 分钟超时自动清除，confirm_id 一次有效
- **OpenAI API Key** 从环境变量 `OPENAI_API_KEY` 读取，不进代码/前端
- 可配置 `OPENAI_BASE_URL` 支持代理或兼容 API（如 Azure OpenAI, 国内代理）

## 6) Vue 前端页面

- **入口**: 在 `AppLayout.vue` 的 header 右侧增加 AI 按钮（闪电 ⚡ 图标），附带 `Ctrl+K` 快捷键提示
- **AI 指令面板**: 使用 Element Plus `<el-dialog>` 浮窗，任意页面可呼出（全局快捷键 Ctrl+K / Cmd+K）
  - 输入框 `<el-input>` type="textarea" :autosize="{ minRows: 1, maxRows: 4 }"
  - Enter 发送（Shift+Enter 换行）
  - 消息列表展示对话历史（用户消息 + AI 回复），用简单的左对齐气泡样式
  - 使用 `v-loading` 显示请求中状态
  - 支持显示 markdown 风格的简单文本（无需额外库，用 `<pre>` 或纯文本展示）
- **确认对话框**: 写入类操作弹出独立 `<el-dialog>`，显示操作摘要 + "确认执行"/"取消" 按钮
  - 点击确认 → 调 `POST /api/ai/confirm` → 显示执行结果
  - 点击取消 → 不调任何接口，仅关闭对话框

**组件文件**: `frontend/src/views/ai/AiCommandPanel.vue`（浮窗组件，由 AppLayout 引用并挂载到全局）

### 与现有前端的集成

- `AppLayout.vue` 在 `<el-header>` 右侧增加按钮，引用 AiCommandPanel
- AiCommandPanel 内部使用 `onMounted` / `onUnmounted` 注册/注销全局键盘事件 `Ctrl+K`

## 7) 离线降级方案

- `ai_service.OfflineAiService` 实现基于关键词 + 正则匹配的简单命令解析
- 关键词示例:
  - `"查产品|搜产品|找产品|有哪些产品"` → `search_product(query=提取的词)`
  - `"查客户|搜客户"` → `search_customer(query=提取的词)`
  - `"送货单|今天|最近.*送货"` → `list_delivery_notes(start_date=today-7)`
  - `"对账单|对账"` → `get_statement(start_date=本月1日, end_date=今天)`
  - `"统计|概况|概览"` → `get_statistics()`
  - `"创建送货单|新建送货单"` → 模糊匹配后返回提示"请详细描述：客户、日期和产品明细"
- 数值/日期提取通过简单正则（`\d+` 匹配数量，`\d{4}-\d{2}-\d{2}` 或自然语言日期）
- 当 OpenAI API 不可用或 API Key 未配置时自动降级
- 降级后在返回的 reply 末尾追加 `"(离线模式)"` 标记
- 写入类操作在离线模式下走规则直接创建，但依然需要二次确认
- 离线模式功能有限（仅支持基础搜索 + 简单查询），复杂语义会回复"暂不支持该指令"

## 9) 实现顺序

1. **后端配置** — `backend/config.py` 新增 OpenAI 配置，`requirements.txt` 追加 `openai`
2. **后端核心** — `backend/services/ai_service.py`（LLM 调用 + function registry + 执行映射 + session 管理 + 离线降级）
3. **后端路由** — `backend/routers/ai.py` + 注册到 `backend/main.py`
4. **前端 API** — `frontend/src/api/ai.js`
5. **前端组件** — `frontend/src/views/ai/AiCommandPanel.vue`
6. **前端集成** — `AppLayout.vue` header 增加入口按钮
7. **依赖安装** — `pip install -r requirements.txt`，`cd frontend && npm install`（前端无新增依赖）
8. **测试验证** — 后端: 验证 LLM 调用、离线降级、确认流程、audit log；前端: 手动验证快捷键、对话流、确认框

- AI 创建/编辑产品、客户（v1.1.0）
